首先来讨论接口型函数的使用：

函数类型实现某一个接口，称之为接口型函数，方便使用者在调用时既能够传入函数作为参数，也能够传入实现了该接口的结构体作为参数。

使用场景:
这种方式适用于逻辑较为复杂的场景，如果对数据库的操作需要很多信息，地址、用户名、密码，还有很多中间状态需要保持，比如超时、重连、加锁等等。
这种情况下，更适合封装为一个结构体作为参数。这样，既能够将普通的函数类型（需类型转换）作为参数，也可以将结构体作为参数，使用更为灵活。

本模块使用接口型函数暴露给用户，使用户可以自行注册函数，在缓存不存在时，调用这个函数，得到源数据。

  • 一个group可以认为是一个缓存的命名空间，每个group有唯一的name；
  • getter Getter，缓存未命中时获取源数据的回调(callback)；
  • mainCache *Cache，一开始实现的并发缓存；

http server

HTTPPool有两个字段：

  • self，记录自己的地址，包括主机名/IP和端口；
  • basePath，节点通讯前缀，默认是/gemmacache/，那么/http://localhost:9999/gemmecache/就是请求的开头；

ServeHTTP的实现逻辑比较简单，首先判断访问路径的前缀是否是 basePath，不是返回错误。
约定访问路径格式是 /basepath/groupname/key，通过 groupname 得到 group 实例，再使用 group.Get(key) 获取缓存数据。
最终使用 w.Write() 将缓存值作为 httpResponse 的 body 返回

分布式缓存

一致性哈希


一致性哈希网上介绍的资料很多，这里只是实现一个简单的哈希环；

Map 是一致性哈希算法的主数据结构，包含 4 个成员变量：

  1. Hash 函数 hash；
  2. 虚拟节点倍数 replicas；
  3. 哈希环 hashCodes；
  4. 虚拟节点与真实节点的映射表 hashMap，键是虚拟节点的哈希值，值是真实节点的名称(地址)；

这里注意两点：

  1. 加入虚拟节点倍数是为了防止节点过少时的数据倾斜问题；
  2. 时刻保证hashCodes有序，查找时要对结果索引取余，来形成环；

同时，在httppool中，增加维护一致性哈希的字段和对其他远程节点连接的字段：

  • hashRing：一致性哈希环，用来实现key到节点的映射；
  • getters：映射远程节点与对应的 httpGetter。每一个远程节点对应一个 httpGetter，因为httpGetter与远程节点的地址 baseURL有关；

增加在group中设置远程节点的方法和根据key查找节点的方法；

  • PickPeer：包装了一致性哈希算法的 Get() 方法，根据具体的 key，选择节点，返回节点对应的 HTTP 客户端；
  • Set：方法实例化了一致性哈希算法，并且添加了传入的节点；

每个节点都包装成一个httpPool，内部维护与其他节点的通信方式，同时实现PickPeer接口。也就是说，HTTPPool 既具备了提供 HTTP 服务的能力，也具备了根据具体的 key，创建 HTTP 客户端从远程节点获取缓存值的能力

最后，在group中增加节点选取的字段。

  • 新增RegisterPeers方法，将实现了 PeerPicker 接口的 HTTPPool 注入到 Group 中；
  • 改造Get接口，当本节点缓存中不存在指定的key时，去查询key应该缓存在哪个节点上，若本就应缓存在本节点，则调用getter查询DB更新缓存，否则去远程节点上获取数据；

优化

防止缓存击穿

  • caller 代表正在进行中，或已经结束的请求。使用 sync.WaitGroup 锁避免重入；
  • ConcurrencyLimiter 是 single_flight 的主数据结构，管理不同 key 的请求(call)，其中的mutex是为了保证并发读写callers线程安全引入的锁；
  • Do 方法，接收 2 个参数，第一个参数是 key，第二个参数是一个函数 f。Do 的作用就是，针对相同的 key，无论 Do 被调用多少次，函数 f 都只会被调用一次，等待 f 调用结束了，返回返回值或错误；

完整代码见:GemmaCache
