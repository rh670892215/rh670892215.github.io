golang的GC

java、c#、golang中都有垃圾回收机制，本文来针对golang的垃圾回收机制进行研究，在研究golang的垃圾回收机制前，先来看一下几种经典的垃圾回收算法。

垃圾回收算法

下面这部分内容在《深入理解java虚拟机》的第三章有详细的讲解，这里只进行简单的介绍。

标记清扫


首先标记当前正在使用的对象，接着清扫掉未标记的垃圾对象；

标记压缩


为了解决内存碎片的问题，在标记清扫算法的基础上进行了升级，在清扫的同时对存活的对象进行压缩整合，使整体空间更为紧凑；

标记复制


标记复制算法流程如下：

  1. 分配两片相等大小的空间，称为 fromspace 和 tospace
  2. 每轮只使用 fromspace 空间，以GC作为分水岭划分轮次
  3. GC时，将fromspace存活对象转移到tospace中，并以此为契机对空间进行压缩整合
  4. GC后，交换fromspace和tospace，开启新的轮次

引用计数

引用计数法优点包括：

  • 原理和实现都比较简单
  • 回收的即时性：当对象的引用计数为 0 时立即回收，不像其他 GC 机制需要等待特定时机再回收，提高了内存的利用率
  • 不需要暂停应用即可完成回收

缺点包括：

  • 无法解决循环引用的回收问题：当 ObjA 引用了 ObjB，ObjB 也引用 ObjA 时，这两个对象的引用次数使用大于 0，从而占用的内存无法被回收
  • 时间和空间成本较高：一方面是因为每个对象需要额外的空间存储引用计数器变量，另一方面是在栈上的赋值时修改引用次数时间成本较高（原本只需要修改寄存器中的值，现在计数器需要不断更新因此不是只读的，需要额外的原子操作来保证线程安全）
  • 引用计数是一种摊销算法，会将内存的回收分摊到整个程序的运行过程，但是当销毁一个很大的树形结构时无法保证响应时间

golang的GC原理

golang在1.8之后，就使用三色标记法+混合写屏障机制来进行gc，首先来看一下三色标记法。

三色标记法

三色标记法原理如下：

  1. 创建三个集合white、grey、black；
  2. 将全部对象放入到white集合中；
  3. 从根节点开始遍历全部对象，将遍历到的对象放入grey集合中；
  4. 遍历grey中对象引用的对象，将当前grey中的对象放入black集合中，将新遍历到的对象，即grey中对象引用的对象放入到grey；
  5. 不断重复4，直到grey中没有任何对象；
  6. white集合中的就是垃圾对象；

在golang中，并不是使用三个集合实现的三色标记法，而是：

  • 白色对象: 某个对象对应的gcMarkBit为0(未被标记)
  • 灰色对象: gcMarkBit为1(已被标记)且在(待scan)gcWork的待scan buffer中
  • 黑色对象: gcMarkBit为1(已被标记)且不在(已经scan)gcWork的待scan buffer中

对于1.5版本前的golang，gc时需要停止全局的用户协程，完成gc后再恢复用户线程，这样会对性能产生较大的影响，因此就需要引入并发的gc。
对于并发，有两个层面的并发：

  1. 标记和清扫的并发；
  2. gc和用户协程的并发；

对于第一种标记和清扫的并发不难实现，因为对象的引用关系不会改变，但是对于第二章，由于用户协程的执行可能会改变对象间的引用关系，下面来看这几个case：
case1:

如上述流程：

  1. 起始root引用A、B，A引用C，B引用D；
  2. A、B标记成grey；
  3. A、B标记成black，D标记成grey，此时A删除对C的引用；
  4. D标记成black，然后D建立对C的引用

这样就导致了D的“漏标”，其核心问题是完成标记的黑对象指向了一个被灰\白对象删除引用的白色对象。

case2:

如上述流程，在第三步中将A标记成black，C标记成grey后，A删除了对C的引用；
这会导致“多标”，即C、D本来是可以被清理的却没有被清理，与“漏标”相比，只是增加了浮动垃圾，在下次gc时就可以被清除了，可以忍受。

插入屏障&删除屏障

下面我们来看一下漏标问题，思考一下死锁问题的解决，死锁的必要条件如下：

  1. 不可剥夺：进程已获得的资源，在未使用完之前，不能强行剥夺；
  2. 请求和保持：一个进程因请求资源而阻塞时，已获得的资源保持不放；
  3. 互斥：一个资源每次只能被一个进程使用；
  4. 循环等待：若干进程之间形成一种头尾相接的循环等待资源关系；

只要破坏其中的任意一条必要条件就可以避免死锁，那么对于漏标问题，我们也同样来分析一下其必要条件。
前面说过其核心问题是完成标记的黑对象指向了一个被灰\白对象删除引用的白色对象，那么进行拆分后其必要条件如下：

  1. 完成标记的黑对象指向了一个白色对象；
  2. 灰\白对象删除对白色对象的引用；
  3. 第一步在第二步之前发生；

有一套强弱三色不变式理论可以解决上述问题：

  • 强三色不变式：白色对象不能被黑色对象直接引用；——破坏必要条件1
  • 弱三色不变式：白色对象可以被黑色对象引用，但要从某个灰对象出发仍然可达该白对象；——破坏1、2条件联动

插入屏障

详细流程在这篇博客中有详细的介绍，这里直说结论。
具体操作：在A对象引用B对象的时候，B对象被标记为灰色。(将B挂在A下游，B必须被标记为灰色)。
对于堆上的对象操作使用插入屏障，而对于栈上的对象不使用，这也就导致会有10ms～100ms的STW来对栈上的对象进行标记。

删除屏障

详细流程在这篇博客中有详细的介绍，这里直说结论。
具体操作：被删除的对象，如果自身为灰色或者白色，那么被标记为灰色。
这种方式的回收精度低，一个对象即使被删除了最后一个指向它的指针也依旧可以活过这一轮，在下一轮GC中被清理掉。

混合屏障

无论是插入屏障、删除屏障都有短板：

  • 插入屏障：结束时需要STW来扫描栈；
  • 删除屏障：回收精度低，GC开始时STW扫描堆栈来记录初始快照，这个过程会保护开始时刻的所有存活对象；

因此在go1.8之后使用混合写屏障，避免了对栈re-scan的过程，极大的减少了STW的时间，其流程如下：

  1. GC开始将栈上的可达对象全部扫描并标记为黑色(之后不再进行第二次重复扫描，无需STW)；
  2. GC期间，任何在栈上创建的新对象，均为黑色；
  3. 被删除的对象标记为灰色；
  4. 被添加的对象标记为灰色；

来看下面这几个case，来逐个分析一下：

  1. 对象被一个堆对象删除引用，成为栈对象的下游
    删除引用时触发删除屏障，将节点置灰；

  2. 对象被一个栈对象删除引用，成为另一个栈对象的下游
    栈对象不启动屏障，但在GC期间新建的对象都会置黑，不会漏标；

  3. 对象被一个堆对象删除引用，成为另一个堆对象的下游
    触发插入屏障，将新对象置灰，不会漏标；

总结：
golang中的混合写屏障满足弱三色不变式，只需要在GC开始的时候扫描各个goroutine的栈，将可达对象置黑并一直保持，整个过程不需要STW，而标记结束后由于栈中的对象是保持黑色的，因此也不需要re-scan，也就减少了STW的时间。
注意混合写屏障在GC期间虽然没有STW，但是在扫描某个具体的栈时，还是要停止这个goroutine赋值器的工作。

GC的触发时机

主动触发：调用runtime.GC；
被动触发：

  1. 通过sysmon监控线程进行监控，自从上次GC后间隔时间达到了runtime.forcegcperiod时间（默认为2分钟），将启动GC；
  2. 使用步调算法，当前分配的内存达到一定阈值时触发，这个阈值在每次GC过后都会根据堆内存的增长情况和CPU占用率来调整；

上面的第二种被动触发，可以通过变量GOGC进行控制，表示最近的一次GC之后，已分配的内存量达到了上次GC后的百分之多少时，触发下一次垃圾回收，默认是100。如：
