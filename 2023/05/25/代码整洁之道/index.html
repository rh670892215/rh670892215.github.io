代码整洁之道

发表于 2023-05-25 分类于 工程素养

有意义的命名：

  1. 名副其实；
  2. 避免误导；
  3. 做有意义的区分；
  4. 使用读得出来的名称；
  5. 使用可搜索的名称；
  6. 避免使用编码（匈牙利语标记法、成员前缀、接口和实现）；
  7. 避免思维映射；
  8. 类名（尽量使用名次或名词短语）；
  9. 方法名（尽量使用动词和动词短语）；
  10. 每个概念对应一个词；
  11. 别用双关语；
  12. 添加有意义的语境（将相关的变量封装到类中，给变量添加语境）；

函数：

  1. 保持函数短小，确保只做一件事，同时让代码读起来像是一系列自顶向下的；
  2. 每个函数一个抽象层级；
  3. 使用工厂模式代替switch语句；
  4. 函数参数尽量少；
  5. 函数要么做什么事，要么回答什么事，二者不可兼得；
  6. 使用异常代替返回错误码；
  7. 抽离try/catch代码块；
  8. 不要重复自己；

格式：

  1. 紧密相关的代码应该相互靠近，适当使用空行进行垂直方向上的区隔；
  2. 相关函数，若某个函数调用了另一个，就应该把他们放在一起，且调用者尽可能放在被调用者上面；

对象和数据结构：

  1. 数据抽象：不要暴露数据细节，而是要以抽象形态表述数据，并不只是使用接口、赋值器、取值器就万事大吉；（对象隐藏数据，暴露操作）
  2. 得墨忒耳定律；

异常：

  1. 使用异常而非返回码；
  2. 先写try-catch-finally语句，尝试编写强行抛出异常的测试，再往处理器中添加行为，是指满足测试要求，就是说要先构造try代码块的事务范围；
  3. 尽量使用不可控异常，若使用可控异常可能会违反开放/闭合原则；
  4. 尽量不要返回null值，不要传递null值；

边界：

  1. 边界上的代码需要清晰的分割和定义了期望的测试，避免我们的代码过多了解第三方代码中的特定信息；

避免将边界的接口（对第三方库的依赖）在系统中传递，否则当第三方库发生改变时可能需要修改很多个地方，应该对第三方库的依赖封装起来，构造整洁的边界，同时要对边界代码做出充分的测试，防止第三方库功能改变的时候影响我们系统的功能；

单元测试：

  1. 快速，测试应该能快速运行；
  2. 独立，测试应该独立，某个测试不应为下一个测试设定条件；
  3. 可重复，测试应当在任何环境中重复通过；
  4. 自足验证，测试应该有布尔值输出，无论通过还是失败，不应该通过查看日志来确认测试是否通过；
  5. 及时，测试应及时编写；

类：

  1. 单一职责原则（SRP），类只应有一个权责——只有一条修改的理由；
  2. 内聚，类应该只有少量实体变量，类中的每个方法都应该操作一个或者多个这种变量，若一个类中的每个变量都被每个方法锁使用，则该类具有最大的内聚性；
  3. 开放——闭合原则（OCP），类应当对扩展开放，对修改封闭；
  4. 依赖倒置原则（DIP），类应当依赖于抽象而不是依赖于具体细节；

迭进：（自上而下优先级递减）

  1. 运行所有测试，遵循有关编写测试不能给持续运行测试的简单、明确的郭泽，系统将会更贴近低耦合度、高内聚度的目标；
  2. 不可重复，可以使用模板方法模式移除高层次重复；
  3. 增强代码的表达力；
  4. 尽可能减少类和方法的数量；

并发：并发是一种策略，它帮助我们把做什么（目的）和何时（时机）做分解开

  • 根据单一职责原则，并发设计自身复杂到成为修改的理由，也应该从代码中分离出来，即分离并发相关代码和其他代码；
  • 谨记数据封装，严格限制对可能被共享的数据的访问；
  • 通过使用数据副本，避免共享数据；
  • 尝试将数据分解到可被独立线程操作到独立子集；
  • 避免使用一个共享对象的多个方法；

1⃣️基于客户端的锁定——客户端代码在调用第一个方法前锁定服务端，确保锁的范围覆盖了调用最后一个方法的代码；

2⃣️基于服务端的锁定——在服务端内创建锁定服务端的方法，调用所有方法，然后解锁。让客户端代码调用新方法；

3⃣️适配服务端——创建执行锁定的中间层。这是一种基于服务端的锁定的例子，但不修改原始服务端代码。

  • 尽可能减小临界区；
  • 尽早考虑关闭问题，尽早另其工作正常；
  • 测试线程代码；

1⃣️不要将错误归咎于偶发事件；

2⃣️不要同时追踪非线程缺陷和线程缺陷，确保代码在线程之外可工作；

3⃣️编写可插拔得线程代码，在不同的配置环境下运行；

4⃣️运行多于处理器数量的线程，任务交换越频繁，越有可能找到问题代码；

5⃣️使用异动策略，使线程以不同的次序执行，可以增加发现错误的机会；

（生产者-消费者模型、读者-作者模型、哲学家模型学习）
