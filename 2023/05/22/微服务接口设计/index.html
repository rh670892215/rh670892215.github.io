微服务接口设计

高可用：

  1. 降级兜底：若依赖的服务挂掉，我们的业务不应该也跟着挂掉，需要制定好兜底的方案；
  2. 过载保护（保护自己）：请求等待超时、服务过载及早拒绝
  3. 流量控制（保护下游）：控制打到下游的流量如漏桶算法、令牌桶算法，必要时需要实现分布式限流；
  4. 快速失败：设置超时时间；
  5. 无状态服务：尽可能使微服务无状态，状态即数据，若某一调用方的请求一定要落到某一后台节点，使用服务在杯底缓存的数据（状态），那么这个服务就是有状态的服务，应该把这些数据迁移到分布式缓存中，这样微服务就能在运行时动态增删节点，不需要考虑缓存数据的同步问题；
  6. 最少依赖：尽可能减少对第三方的依赖；
  7. 简单可靠：可靠性只有靠不断追求最大程度的简化而得到；
  8. 分散原则：可以基于主次进行服务拆分，将重要接口放到一个服务中，次要接口放到另外一个服务中；
  9. 隔离原则：不同模块间相互隔离，如部署隔离、DB隔离；
  10. 幂等设计：对接口的多次调用所产生的结果和调用一次是一致的；
  11. 故障自愈：故障不可避免，系统要有自愈能力，如消息处理或异步逻辑等非关键操作失败引发的数据不一致，需要有最终一致性的操作，如兜底的定时任务、失败重试队列、用户在下次请求时触发修复逻辑；
  12. CAP定理：一致性（consistency）、可用性（availability）、分区容错性（partition tolerance），三者不能兼顾，必须在一致性（C）和可用性（A）做出取舍，若服务偏业务逻辑，对接用户，应选择AP，遵循BASE理论，若服务偏控制，对接服务，应选择CP，遵循ACID理论；
  13. BASE理论：
    • 基本可用（Basically Available）：服务降级兜底；
    • 软状态（Soft state）：允许系统在多个不同节点的数据副本存在数据延时；
    • 最终一致性（Eventual consistency）：软状态应有时间期限，期限过后应当保证所有副本保持数据一致性；

<div><span style="font-family: unset;">                        （3）回写模式，写操作时只更新缓存并立即返回，然后异步写SoR；</span></div>

    <li>
        <div>缓存淘汰策略：（1）基于时间；（2）基于空间；（3）基于容量；（4）基于引用；</div>

    <li>
        <div>淘汰算法：FIFO、LRU、LFU</div>

    <li>

<h2>存储设计：</h2>
<ul>
    <ul>
        <li>
            <div>读写分离：</div>
        </li>
        <ul>
            <li>
                <div>特点：（1）数据库服务是主从架构（2）主节点负责写操作，从节点负责读操作（3）主节点将数据复制到从节点</div>
            </li>
            <li>
                <div>避免复制滞后带来的问题的方法：</div>
            </li>
            <ul>
                <li>
                    <div>写后读一致：即读自己的写，适用于用户写操作后要求实时看到更新；</div>
                </li>
                <li>
                    <div>二次读取：优先读取从节点，若读取失败或跟踪的更新时间小于某个阈值，则再从主节点读取；</div>
                </li>
                <li>
                    <div>区分场景：关键业务读取主节点，非关键业务读写分离；</div>
                </li>
                <li>
                    <div>单调读：保证用户的读请求都发到同一个从节点，避免出现回滚的现象；</div>
                </li>
            </ul>
        </ul>
        <li>
            <div>分库分表：</div>
        </li>
        <ul>
            <li>
                <div>需要分库分表的情况：（1）单表的数据量达到了一定的量级（mysql一般是千万级）；（2）数据库吞吐量达到瓶颈</div>
            </li>
            <li>
                <div>切分模式：</div>
            </li>
            <ul>
                <li>
                    <div>垂直切分：</div>
                </li>
                <li>
                    <div>水平切分：</div>
                </li>
            </ul>
        </ul>
        <li>
            <div>动静分离： 将经常更新的·数据和更新频率低的数据进行分离，有点像垂直切分，将动态和静态的字段分别存储在不同的库表中，减小数据库锁的粒度；</div>
        </li>
        <li>
            <div>冷热分离：</div>
        </li>
        <li>
            <div>重写轻读：冗余写，减轻读的压力，如写扩散；</div>
        </li>
        <li>
            <div>数据异构：存储不同结构的数据，如数据格式的异构、数据存储地点的异构、数据存储逻辑的异构（如按照不同的索引维度来存储同一份数据）</div>

<h2>易维护：</h2>
<ol>
    <li>
        <div>充分必要：不是随便一个功能就要有个接口；</div>
    </li>
    <li>
        <div>单一职责：每个API只专注做一件事；</div>
    </li>
    <li>
        <div>内聚接偶：一个接口要包含完整的业务功能，不同接口之间的关联要尽可能的小（一般使用MQ完成接口间的耦合）；</div>
    </li>
    <li>
        <div>开闭原则：对扩展开放，对修改关闭；</div>
    </li>
    <li>
        <div>统一原则：接口要具备统一的命名规范、统一的出入参风格、统一的异常处理流程、统一的错误码定义、统一的版本规范等；</div>
    </li>
    <li>
        <div><span style="font-size: 13.000000953674316px;">用户重试：接口失败时，应该尽可能的由用户重试，而不是由后台进行处理；</span></div>
    </li>
    <li>
        <div><span style="font-size: 13.000000953674316px;">最小惊讶：代码应该尽可能避免读者蒙圈；</span></div>
    </li>
    <li>
        <div><span style="font-size: 13.000000953674316px;">避免无效请求：不要传递无效请求至下游；</span></div>
    </li>
    <li>
        <div><span style="font-size: 13.000000953674316px;">入参校验：做好入参校验，及早发现无效请求并拒绝；</span></div>
    </li>
    <li>
        <div><span style="font-size: 13.000000953674316px;">设计模式：适当使用设计模式；</span></div>
    </li>
    <li>
        <div><span style="font-size: 13.000000953674316px;">禁用flag标志：</span></div>
    </li>
    <li>
        <div><span
                style="font-size: 13px;">页宜小不宜大：结果集包含成千上百条数据时，返回查询就需要分页，通常通过一个offset偏移量或者页码进行分页，然后通过API一页一页的查询；</span>
        </div>
    </li>
</ol>

<h2><span style="font-size: 13.000000953674316px;">防止sql注入：</span></h2>

    <ul>
        <li>
            <div>严格检查输入变量的类型和格式；</div>
        </li>
        <li>
            <div><span style="font-size: 13.000000953674316px;">过滤和转义特殊字符；</span></div>
        </li>
        <li>
            <div><span style="font-size: 13.000000953674316px;">使用参数化查询，而非手动拼接SQL；</span></div>
        </li>
    </ul>
</ul>
<h2><span style="font-size: 13.000000953674316px;">防刷：</span></h2>
<div>基于用户ID、基于IP、基于设备ID防刷</div>
<h2><span style="font-size: 13.000000953674316px;">防篡改：（使用了https也不能保证绝对安全）</span></h2>
<ul>
    <ul>
        <li>
            <div>客户端使用约定好的密钥对传输的参数进行加密，得到签名值s1，一般用HMAC；</div>
        </li>
        <li>
            <div><span style="font-size: 13.000000953674316px;">客户端将签名放入请求，发送给服务器；</span></div>
        </li>
        <li>
            <div><span style="font-size: 13.000000953674316px;">服务器收到请求，使用约定好的密钥对请求的参数再次进行加密，得到签名值s2；</span></div>
        </li>
        <li>
            <div><span style="font-size: 13.000000953674316px;">服务器比对s1和s2，若不一致则说明参数被篡改；</span></div>
        </li>
    </ul>
</ul>
<h2>防重放：</h2>
<ul>
    <ul>
        <li>
            <div>重放：恶意用户抓取真实的接口请求包，不提供的发送重复请求；（对写接口影响较大）</div>
        </li>
        <li>
            <div><span style="font-size: 13.000000953674316px;">解决方法：</span></div>
        </li>
        <ul>
            <li>
                <div><span style="font-size: 13.000000953674316px;">客户端基于“请求内容+时间戳+密钥”计算一个签名s1，一般使用HMAC；</span>
                </div>
            </li>
            <li>
                <div><span style="font-size: 13.000000953674316px;">客户端请求后台接口时带上s1；</span></div>
            </li>
            <li>
                <div><span
                        style="font-size: 13.000000953674316px;">后台拿到签名后，使用心啊沟通的算法计算出一个签名与s1比较，若不一致则非法，直接拒绝；</span>
                </div>
            </li>
        </ul>
    </ul>
</ul>
